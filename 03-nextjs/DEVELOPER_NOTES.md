# V3 — Next.js 15 Static Export Bundle

**Audience:** Clients with an existing React / Next.js codebase, or who may want to extend the IR deliverable into a logged-in investor portal, interactive dashboards, or filings search in the future.
**Goal:** A fully-static IR page (no Node runtime at serve time) that fits naturally into a React ecosystem. Ship the `out/` folder as a zip; integrators can also import individual components directly.

---

## What this is

- **Next.js 15 (App Router)** with `output: 'export'`.
- React 18 Server Components — content is rendered to HTML at build time, hydrated on the client for tabs + mobile nav.
- Two root layouts via route groups for per-locale `<html dir>` control:
  - `app/(en)/` → `/` (LTR, lang="en")
  - `app/(ar)/ar/` → `/ar/` (RTL, lang="ar")
- Same component contract as the Astro build — components receive `copy` props and render semantic HTML.

## How to build and zip

```bash
npm install
npm run build
# out/ is the shippable folder.
# Windows:  powershell Compress-Archive -Path out/* -DestinationPath ir-nextjs.zip
# macOS/Linux: zip -r ir-nextjs.zip out/
```

The receiving client unzips the `out/` folder and deploys to any static host.

**If a client wants to integrate individual components** (e.g., a Next.js site that already has its own chrome): zip the `components/` and `lib/` folders along with `app/global.css`. The integrator copies them into their tree, imports the components they need, and wires up the widget loader.

## How widgets are integrated

Same contract as V2/V4. `public/widgets.js` is shipped to `out/widgets.js` at build. Each `<div data-widget="...">` gets lazy-mounted via IntersectionObserver + the third-party `widget-loader.js` from `irp.atnmo.com`.

UUID / listingId / version configured in `public/widgets.js`:

```js
uuid:      '5be9c146-613e-4141-a351-1f5e13fc5513',
listingId: '81a06c05-1a48-4d1b-8dbd-bcf60a76730f',
version:   'v3',
```

To add a new widget: edit `lib/widgets.js` under `TABS`, add a tab label in `TAB_LABELS`, rebuild. `WidgetTabs.jsx` renders the table.

## Bilingual (EN / AR)

Two route groups, two `<html>` declarations:

- `/` → EN, `<html lang="en" dir="ltr">`
- `/ar/` → AR, `<html lang="ar" dir="rtl">`

Copy centralised in `lib/widgets.js`. To add a third locale, create `app/(fr)/fr/page.jsx` + a new layout + extend the `COPY` / `CASE_ITEMS` / `TAB_LABELS` dictionaries.

For a more scalable locale system, swap to `next-intl` (adds ~10 KB but gives pluralization, date formatting, etc.).

## Benchmark snapshot

See `BENCHMARK.md`. Headline: Mobile Lighthouse 67, Desktop 98. The mobile score is lower than V2/V4 because of the React runtime (~100 KB) — a real trade-off for the ergonomics/ecosystem. If a client values mobile Lighthouse over React DX, ship V2 Astro instead.

## Gotchas

- **Two root layouts.** Next.js requires exactly one root layout per route. By using `(en)/` and `(ar)/` route groups, each gets its own root layout with a different `<html dir>`. This is the documented pattern.
- **`app/layout.jsx` is NOT at the root.** Each route group has its own layout. If you add a global page like `/contact`, decide which locale tree it belongs in (or create a third route group).
- **`images.unoptimized: true`** in `next.config.mjs` because static export doesn't support the `next/image` optimizer. If you need optimized images, move to a Node-rendered deployment.
- **Theme-color / metadata:** Use the new Next 15 `viewport` export for `themeColor`. Don't put it in the legacy `metadata` — Next warns.
- **Hydration warnings on RTL:** Make sure no component reads `window`/`document` during SSR. Any "use client" directive is acceptable if a component truly needs browser APIs (none of the shipped components do).
- **Widgets are third-party.** If they break, it's upstream at `irp.atnmo.com`.
- **CLS of 0.181** — same as V2/V4; fixable by reserving heights in CSS. Optional.

## Extending for other clients

When to pick V3 over V2:
- Client already uses React / Next.js → they can drop components directly into their app.
- Future roadmap needs interactive islands (filings search, investor login, portfolio tools).
- Team velocity with React > Lighthouse-score margin.

Per-client workflow:
1. Clone this folder.
2. Edit `CONFIG` in `public/widgets.js` (UUID, listingId).
3. Edit `COPY` / `CASE_ITEMS` in `lib/widgets.js`.
4. Replace `public/img/logo.svg` and hero image.
5. Adjust brand tokens at the top of `app/global.css`.
6. `npm run build`. Ship `out/`.

If a client requests a logged-in investor portal: keep this static IR page for public pages and add `app/portal/` with client-side rendered components, switched to a Node runtime (remove `output: 'export'`). The widget loader continues to work in both modes.
