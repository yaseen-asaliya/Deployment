# V2 вЂ” Astro Static Site Bundle

**Audience:** Clients with a Jamstack / static-hosting deployment (Netlify, Cloudflare Pages, Vercel, S3 + CloudFront, GitHub Pages) or green-field projects where payload-size and SEO matter most.
**Goal:** Ship a pre-rendered IR page with the smallest possible JS payload вЂ” just enough for tab switching, mobile-nav toggle, and widget lazy-mount.

---

## What this is

- **Astro 6** in static (`output: 'static'`) mode.
- Pure HTML output at build time. The only JS shipped is the ~4 KB `widgets.js` that lazy-mounts the IR widgets.
- Two pages: EN (`/`) and AR (`/ar/`) вЂ” each is a single HTML file generated at build time.
- No client-side framework. No React, no Vue, no hydration. Astro's `.astro` components render to HTML on disk.

## How to run locally

```bash
npm install
npm run dev        # Astro dev server at http://localhost:4321
npm run build      # Generate dist/
npm run preview    # Serve dist/ at http://localhost:4321
```

Build + serve without Astro CLI:

```bash
npm run build
npx http-server dist -p 8002 -c-1
```

## How to build and zip

```bash
npm install
npm run build
# dist/ is the shippable output.
# To zip for client handoff:
#   Windows:  powershell Compress-Archive -Path dist/* -DestinationPath ir-astro.zip
#   macOS:    zip -r ir-astro.zip dist/
#   Linux:    zip -r ir-astro.zip dist/
```

The client unzips, deploys the folder to any static host, and the IR page is live. The dev tooling (node_modules, astro, etc.) is not needed at runtime.

## How widgets are integrated

Identical contract to V4 Plain HTML. In any component you can write:

```astro
<div class="widget">
  <div class="widget__header"><h3 class="widget__title">Share Price</h3></div>
  <div class="widget__body" data-widget="share-price"></div>
</div>
```

`public/widgets.js` finds every `[data-widget]`, lazy-mounts via IntersectionObserver (500 px root margin), and invokes the third-party `loadWidget(widgetType, UUID, lang, listingId, 'v3')` once the widget-loader.js from `irp.atnmo.com` is ready. The `<html lang>` attribute determines the widget locale.

UUID / listingId / version are configured in `public/widgets.js`:

```js
uuid:      '5be9c146-613e-4141-a351-1f5e13fc5513',
listingId: '81a06c05-1a48-4d1b-8dbd-bcf60a76730f',
version:   'v3',
```

To add a new widget: add an entry in `src/data/widgets.js` under `TABS`, add its label to `TAB_LABELS`, and re-build. `WidgetTabs.astro` iterates the data and emits the tab + panel automatically.

## Bilingual (EN / AR)

Two routes, one shared component tree:

- `src/pages/index.astro` в†’ `dist/index.html` (EN, `dir="ltr"`)
- `src/pages/ar/index.astro` в†’ `dist/ar/index.html` (AR, `dir="rtl"`)

Copy lives in `src/data/widgets.js` under `COPY.en` and `COPY.ar`. Widget tab labels in `TAB_LABELS.en` and `TAB_LABELS.ar`.

To add a third locale:
1. Add a `COPY.fr`, `TAB_LABELS.fr`, `CASE_ITEMS.fr` block in `src/data/widgets.js`.
2. Create `src/pages/fr/index.astro` passing `lang="fr"`.
3. Re-build.

## Cache & hosting

For Netlify, drop `public/_headers`:

```
/_assets/*
  Cache-Control: public, max-age=31536000, immutable
/widgets.js
  Cache-Control: public, max-age=604800
/fonts/*
  Cache-Control: public, max-age=31536000, immutable
/img/*
  Cache-Control: public, max-age=604800
/*.html
  Cache-Control: no-cache
```

For Cloudflare Pages: use the same `_headers` convention. For Vercel: use `vercel.json`. For nginx/Apache: translate to `expires` / `Header set Cache-Control`.

## Gotchas

- **Astro 6 dropped the `--legacy-peer-deps` need** but double-check `node --version` is в‰Ґ 20.
- **Fonts are served from `/fonts/`** вЂ” the `@font-face` URL is absolute. Don't put the page at a sub-path without adjusting `base:` in `astro.config.mjs`.
- **Widgets are third-party.** If a widget fails, check the browser console; the issue is upstream at `irp.atnmo.com`.
- **CLS of 0.181** is from the lazy iframe mount changing document height. If Lighthouse CLS matters: set a `min-height` on each `.widget__body` matching its expected iframe height (requires knowing each widget's typical height вЂ” they post it back via `postMessage({type: 'widget-height'})`).
- **The dev server at `astro dev` will NOT test widget behavior accurately** because HMR + widgets.js timing can differ. Always test against `astro build` + `astro preview` (or `http-server dist`) before shipping.

## Extending for other clients

This stack is the recommended choice for clients who want:
- A standalone IR microsite.
- Best-in-class Lighthouse and SEO.
- Pure static hosting (no Node runtime at serve time).
- Minimum JS footprint.

Per-client workflow:
1. Clone this folder.
2. Edit `CONFIG` in `public/widgets.js` (UUID, listingId).
3. Edit `COPY` / `CASE_ITEMS` in `src/data/widgets.js`.
4. Replace `public/img/logo.svg` and hero image.
5. Adjust brand tokens at the top of `src/styles/global.css`.
6. `npm run build`. Ship `dist/`.

If a client needs a third locale: add it to the `COPY` dictionary and add a page file under `src/pages/<locale>/`.

If a client requests interactive islands (e.g., a filterable filings table, a login-only page): add `@astrojs/react` or `@astrojs/solid-js` and use Astro's `client:visible` directive. Keep the rest static.
